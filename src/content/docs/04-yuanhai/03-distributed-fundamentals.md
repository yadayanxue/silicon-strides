> 光速有限，故障总在。

单机数据库的 ACID 事务基于一个隐含假设：所有数据在一台机器上，延迟可忽略，故障是特例。**分布式系统**推翻了这个假设——数据分散在多台机器上，网络消息可能延迟、可能乱序、可能永远不回来。2000 年，Eric Brewer 提出了**CAP 定理**，将这一本质困境浓缩为三个字母：一致性、可用性和分区容错——三者只能选其二。

本章从 CAP 定理出发，走完一致性模型的完整光谱（从强一致性到最终一致性），深入向量时钟解决冲突的数学原理，最后剖析分布式事务的 2PC/3PC/TCC 协议。

---

## CAP 定理：三者只能选其二

| 属性 | 含义 | 违背时的表现 |
|------|------|------------|
| **Consistency**（一致性） | 所有节点在同一时刻看到相同的数据 | 读出的数据是旧的 |
| **Availability**（可用性） | 每个请求都能得到非错误的响应 | 请求超时或返回错误 |
| **Partition Tolerance**（分区容错） | 系统在网络分区下继续运行 | 分区两侧无法通信 |

CAP 的关键洞察：**网络分区必然发生**，所以 P 必须被选择。真正的取舍在 C 和 A 之间——选择 C（如 HBase、ZooKeeper）意味着分区期间牺牲可用性；选择 A（如 Cassandra、DynamoDB）意味着分区期间接受弱一致性。

:::note[CAP 的微妙之处]
"CAP 三者只能选其二"是简化版本。更精确的说法是：网络分区发生时，系统必须在一致性和可用性之间选择。分区恢复后，系统可以重新满足两者。日常运行中（无分区），系统可以同时提供 C 和 A。
:::

## 一致性模型光谱

一致性不是二元的——"强"和"最终"之间有一整个光谱：

| 模型 | 保证 | 典型系统 |
|------|------|---------|
| **线性一致性**（Linearizability） | 所有操作看起来在某个全局时间点原子执行 | etcd, ZooKeeper |
| **顺序一致性**（Sequential） | 每个节点看到相同的操作顺序，但不保证实时顺序 | 分布式共享内存 |
| **因果一致性**（Causal） | 有因果关系的操作在所有节点看到相同顺序 | MongoDB（因果一致会话） |
| **最终一致性**（Eventual） | 如果没有新更新，所有副本最终一致 | DNS, Cassandra |

## 向量时钟：冲突检测的数学工具

当两个节点独立更新同一个数据项时，简单的版本号无法判断冲突。**向量时钟**将每个节点的计数器组合为一个 N 维向量，精确定义"发生在先"关系：

```
节点A: [A:2, B:1, C:0]
节点B: [A:2, B:2, C:0] → B 的时钟在向量上"大于"A（B 看到了 A 的所有更新再加自己的更新）
节点C: [A:2, B:0, C:1] → 与 A 的时钟"并发"（两者各有对方没看到的更新）
```

向量时钟的判断规则：

$$
X \prec Y \iff \forall_i (X_i \leq Y_i) \land \exists_j (X_j < Y_j)
$$

如果既不 $X \prec Y$ 也不 $Y \prec X$，则 X 和 Y **并发**——应用层需要解决冲突（CRDT、Last-Write-Wins、或合并函数）。

## 分布式事务：2PC/3PC/TCC

**两阶段提交**（2PC，Two-Phase Commit）是最经典的分布式事务协议：

1. **Prepare 阶段**：协调者问所有参与者"你能提交吗？"参与者回复 Yes/No
2. **Commit 阶段**：如果所有参与者都回复 Yes，协调者发出 Commit；否则发出 Abort

2PC 的致命缺陷是**协调者单点故障**——如果协调者在 Commit 阶段崩溃，已回复 Yes 的参与者被"阻塞"，不知道应该提交还是回滚。**三阶段提交**（3PC）增加了预提交阶段以减少阻塞窗口，但不能完全消除。**TCC**（Try-Confirm-Cancel）通过业务层补偿替代数据库层回滚，是微服务架构的分布式事务主流方案。
